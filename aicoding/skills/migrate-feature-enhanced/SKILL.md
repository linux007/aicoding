# 模块接口功能迁移 Skill（增强版）

## Concept & Trigger

**触发场景：** 当你需要将某个功能从老模块迁移到新模块实现时使用。

**典型场景：**
- 新模块需要实现老模块已有的某个功能
- 功能迁移：老模块 A 的功能 → 新模块 B
- PHP 老实现 → Go 新实现，或 Go 旧实现 → Go 新实现
- 需要复用老模块的数据源和业务逻辑

## 核心原则

### 1. 逐行等价直译（禁止自主重构）

```
✓ 必须做：
  - 逐行翻译老模块的每一行逻辑
  - 保持相同的变量命名逻辑、条件判断顺序、循环遍历顺序、异常处理分支

✗ 禁止做：
  - 禁止 AI 自主重构代码风格
  - 禁止简化复杂的条件嵌套
  - 禁止合并重复逻辑（除非显式要求）
  - 禁止改变算法实现方式
  - 禁止脑补缺失的依赖关系
```

**为什么重要：** 迁移的目标是 100% 等价，而非代码优化。自主重构会导致业务逻辑偏差。

### 2. 分层分步迁移

遵循"由浅入深、逐层验证"原则，严格按顺序执行：

```
1. 外层入口   → 先迁移调用入口点
2. 中层业务   → 再迁移业务处理逻辑
3. 内层子方法 → 然后迁移被调用的子方法
4. 底层工具   → 最后迁移工具函数

每完成一层，必须 Diff 验证后再进入下一层。
```

## Workflow

### Phase 1: 迁移前期规划

**必须完成以下步骤才能进入 Phase 2：**

1. **Brainstorming** — 使用 `superpowers:brainstorming` 理解迁移目标
2. **创建/检查 OpenSpec** — 使用 `/opsx:new` 或检查现有 `spec.md`
3. **代码探索** — 按优先级使用 LSP/Explore/Grep/Read 工具调研老模块实现
4. **抽取核心业务契约** — 记录到 spec.md

**Phase 1 关键输出（记录到 spec.md）：**
- 源模块接口契约
- 调用链路层级图：入口 → 业务处理 → 子方法 → 工具函数
- 依赖关系表
- 隐式依赖清单（全局状态、side effects）

### Phase 2: 实现

1. **分层指令迁移** — 按「核心原则 2」的分层顺序执行，禁止自主重构
2. **TDD 循环** — 使用 `superpowers:test-driven-development`
3. **Diff 验证** — 每次迁移完成必须执行，对比老模块和新模块的对应逻辑
4. **更新 spec.md** — 使用 `/opsx:continue` 记录实现决策

### Phase 3: 验证与收尾

1. **一致性验证保障**
   - 复用老模块单元测试用例
   - 覆盖正常、异常、边界全场景
   - 执行全量回归校验

2. **Code Review** — 使用 `superpowers:requesting-code-review`

3. **风险与质量管控** — 预判隐式风险，制定回滚方案

## 工具优先级

| 工具 | 适用场景 |
|------|----------|
| LSP hover | 查看 struct 定义、类型信息 |
| LSP goToDefinition | 跳转方法实现 |
| LSP findReferences | 查找符号引用 |
| Agent Explore | 未知区域探索（PHP/Go 混合） |
| Grep | 精确文本匹配 |
| Read | 已知路径文件（最后备选） |

## 强制规则

- [ ] Phase 1 每个步骤必须完成才能进入 Phase 2
- [ ] 未完成 brainstorming 禁止修改任何代码
- [ ] 未创建 OpenSpec 禁止进入实现阶段
- [ ] 迁移必须按分层顺序执行，禁止自主重构
- [ ] 每层 Diff 验证通过后才能进入下一层

## 验收标准

- [ ] 功能逻辑已迁移，与源模块行为一致
- [ ] 接口契约保持不变
- [ ] 遵循目标模块架构和风格
- [ ] 相关测试通过
- [ ] spec.md 已更新
- [ ] Code review 通过
- [ ] 全量回归测试通过

## 不要做

- 不要改变现有接口契约（除非明确要求）
- 不要直接复制老模块代码到新模块
- 不要破坏老模块的现有功能
- 不要忽略目标模块的架构风格
- 不要自主重构代码
- 不要忽略隐式依赖（全局变量、side effects）
- 不要跳过 Diff 验证步骤
- 不要自动进行 git 提交
