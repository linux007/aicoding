## 二、流程选择器

### 场景 → 流程映射

| 场景 | 流程 | OpenSpec 使用 | Superpowers 使用 |
|------|------|--------------|-----------------|
| **新功能开发** | Flow A 完整流程 | ✓ 全量使用 | ✓ TDD + Review |
| **Bug 修复** | Flow B 调试流程 | ✓ 探索 + 记录 | ✓ Debug + TDD |
| **小需求修改** | Flow C 快速流程 | ✗ 跳过 | ✓ 简化 TDD |
| **代码重构** | Flow D 规划流程 | ✓ 变更记录 | ✓ TDD + Review |
| **代码迁移** | Flow D 规划流程 | ✓ 迁移规格 | ✓ TDD + Verify |

### 判断规则

```markdown
IF 影响范围 > 3 个文件 OR 需要团队评审 OR 需要历史追溯:
    使用 OpenSpec
ELSE:
    跳过 OpenSpec

IF 涉及业务逻辑代码:
    使用 Superpowers TDD
ELSE:
    可简化流程
```

### 检查清单

```markdown

- [ ] design.md 包含架构图、数据流
- [ ] tasks.md 任务粒度适中（每任务 1-2 小时）

## Phase 2 检查点
- [ ] 计划与 tasks.md 一一对应

- [ ] 每个任务有测试验证命令

## Phase 3 检查点
- [ ] 所有测试先行（RED 验证）
- [ ] 覆盖率 ≥ 80%
- [ ] 无 CRITICAL/HIGH 问题

## Phase 4 检查点
- [ ] specs 已更新
- [ ] archive 目录已创建
- [ ] 临时文件已清理
```

### 判断是否需要测试

| 修改类型 | 是否需要测试 | 原因 |
|---------|-------------|------|
| 业务逻辑 | ✓ 需要 | 防止回归 |
| 配置修改 | ✗ 不需要 | 手动验证即可 |
| 文案修改 | ✗ 不需要 | 无逻辑风险 |
| 日志添加 | ✗ 不需要 | 不影响业务 |
| 样式调整 | ✗ 不需要 | 视觉验证 |
| 常量修改 | ✗ 不需要 | 编译验证 |

### 检查清单

```markdown
## 执行前检查
- [ ] 确认符合"小需求"定义
- [ ] 不符合则切换到 Flow A 或 Flow B

## 执行后检查
- [ ] 手动验证完成
- [ ] Commit message 清晰
```

### 重构 vs 迁移的区别

| 维度 | 重构 | 迁移 |
|------|------|------|
| **目标** | 改善内部结构 | 从 A 系统迁移到 B 系统 |
| **行为** | 必须保持不变 | 可能有变化 |
| **specs** | 定义"不变"的验证 | 定义"等效"映射 |
| **风险** | 引入 bug | 数据丢失、行为偏差 |
| **验证** | 测试不变 | 需要对比验证 |

### 迁移特殊步骤

```markdown
## 迁移额外检查

### Phase 1 额外
- [ ] 源系统行为已记录
- [ ] 目标系统约束已明确
- [ ] 数据映射关系已定义

### Phase 3 额外
- [ ] 并行运行期验证（双写双读）
- [ ] 数据一致性检查
- [ ] 回滚方案已测试

### Phase 4 额外
- [ ] 迁移报告已生成
- [ ] 数据完整性验证
- [ ] 监控指标正常
```

### 检查清单

```markdown
## Phase 1 检查点
- [ ] proposal.md 包含重构/迁移动机
- [ ] specs/ 定义行为保持点
- [ ] design.md 包含新结构设计
- [ ] tasks.md 分步粒度小（每步可独立验证）

## Phase 2 检查点
- [ ] 现有测试全部通过
- [ ] 关键路径有测试覆盖
- [ ] 行为快照已记录

## Phase 3 检查点
- [ ] 每步提交后测试通过
- [ ] 无大爆炸式修改
- [ ] 随时可回滚

## Phase 4 检查点
- [ ] 全量测试通过
- [ ] 覆盖率 ≥ 重构前
- [ ] 性能无退化
```

## 七、场景快速选择指南

### 决策树

```
用户需求
    │
    ├─ 是否是新功能？
    │       └─ 是 → Flow A (新功能开发)
    │
    ├─ 是否是 Bug？
    │       └─ 是 → Flow B (Bug 修复)
    │
    ├─ 修改量 < 20 行 且 无需归档？
    │       └─ 是 → Flow C (小需求修改)
    │
    ├─ 是否改结构不改行为？
    │       └─ 是 → Flow D (重构)
    │
    └─ 是否从 A 系统迁移到 B 系统？
            └─ 是 → Flow D (迁移)
```

### 场景对照表

| 场景 | 流程 | OpenSpec | Superpowers | 归档 |
|------|------|----------|-------------|------|
| 新功能开发 | Flow A | ✓ propose | ✓ TDD | ✓ |
| Bug 修复 | Flow B | ✓ propose | ✓ debug+TDD | ✓ |
| 小需求修改 | Flow C | ✗ | ✗/简化 TDD | ✗ |
| 代码重构 | Flow D | ✓ propose | ✓ TDD | ✓ |
| 代码迁移 | Flow D | ✓ propose | ✓ TDD+验证 | ✓ |
